# Milestone 7 — Implement the Core Combat Resolver

Create the standalone mechanical engine that Battle will use later.

### Resolver responsibilities

```python
CombatResolver(rng=random).resolve_move(actor, target, move_name)
```

The resolver takes a move name, retrieves the exact canonical `Move` from the
actor's current structured move roster, and never executes an arbitrary external
move object. Duplicate move names in one roster are malformed.

Validation and mutation order:

1. Validate actor.
2. Validate target.
3. Resolve the canonical actor-owned move.
4. Validate move kind.
5. Validate move mechanic.
6. Validate target type.
7. Validate resource affordability.
8. Spend the move resource.
9. Roll accuracy.
10. Resolve hit or miss.
11. Apply damage or healing on hit.
12. Apply Super generation for accepted non-Super actions.
13. Return `MoveResult`.

### Formula philosophy

The first formulas should be:

* transparent
* easy to test
* easy to rebalance
* shared across characters
* free of character-specific branching

Do not bury named-character exceptions inside the generic resolver.

Locked provisional formulas:

```text
scaling = floor(arithmetic mean of effective scaling stats)
raw damage = move.power + scaling
physical mitigation = target Constitution // 4
magical mitigation = target Spirit // 4
hybrid defense = floor((target Constitution + target Spirit) / 2)
hybrid mitigation = hybrid defense // 4
final damage = max(1, raw damage - mitigation)
healing = move.power + scaling
```

All scaling and defense stats are read through `combatant.effective_stat()`.
Damage and healing mutate the existing `Health` resource and `MoveResult`
reports only the actual HP removed or restored after clamping.

### Randomness

Inject the random-number source:

```python
CombatResolver(rng=random)
```

Tests should supply deterministic RNG behavior.

The resolver always rolls exactly once for accepted actions:

```python
roll = rng.randint(1, 100)
hit = roll <= move.accuracy
```

### First supported actions

* basic physical damage
* basic magical damage
* healing
* mana spending
* accuracy and misses
* defense mitigation
* self-targeting
* enemy-targeting
* persistent Super spending and generation

Supported mechanics are deliberately narrow:

```text
None
basic_attack
heavy_attack
```

Specialized mechanics such as stagger, burn, shock, water, lightning, storm,
staff control, multi-hit behavior, critical bonuses, and character-specific
setup or payoff rules are rejected as `unsupported_mechanic` until later
milestones. Utility moves are rejected as `unsupported_move_kind`.

Target semantics are intentionally minimal until encounter sides exist:

* `SELF` requires `target is actor`.
* `ENEMY` requires `target is not actor`.

Resource behavior:

* rejected actions spend nothing, roll no accuracy, mutate no health, and
  generate no Super
* accepted actions spend their resource before accuracy
* accepted misses keep their resource cost
* accepted non-Super actions gain 10 persistent Super after resolution
* Super actions spend persistent Super and generate no Super
* Super remains outside `CombatState`

Current M8/M9 dependency:

The resolver is mechanically complete for its approved Milestone 7 boundary, but
rewiring Battle in Milestone 8 cannot yet provide a complete playable
structured-move loop for all four Drifters without either implementing
additional generic mechanic behavior or completing the relevant character-kit
work.

### Gate

The resolver can execute a complete combat action without:

* printing
* reading input
* knowing which character profile is active
* knowing which story scene is active
* rewiring the interactive Battle loop

# Milestone 7.7 - Core Defend Contract

Add Defend as a core combat action without rewiring Battle.

Defend is not an authored `Move`. `CombatResolver.resolve_defend(actor,
combat_state)` validates the actor, validates encounter state, activates a
temporary defending entry in `CombatState`, grants the existing accepted
non-Super-action 10 Super when the actor can generate Super, and returns a
`MoveResult`.

`CombatResolver.resolve_move(actor, target, move_name, *, combat_state=None)`
continues to resolve canonical actor-owned moves. When a valid `CombatState` is
provided and the target is defending, Defend applies after the normal Milestone
7 damage and mitigation formula.

Defend reduction floors the reduction amount and keeps minimum damage at 1:

```text
reduction_amount = normal_damage * reduction_percent // 100
defended_damage = max(1, normal_damage - reduction_amount)
```

Player Defend uses:

```text
physical = min(70, 30 + effective Strength)
magical = min(60, 20 + effective Spirit)
hybrid = min(50, 10 + effective Dexterity)
```

Enemy Defend uses:

```text
physical = 50
magical = 40
hybrid = 30
```

Enemy Defend requires authored `EnemyCapability.DEFEND`. Ordinary Goblins do
not have that capability.

Defend state is temporary encounter state owned by `CombatState`, tracked by
combatant identity, and never snapshotted or persisted. Expiration happens only
through accepted-action completion:

```python
combat_state.complete_accepted_action(actor, opposing_combatants)
```

Accepted hits, misses, healing, and Defend consume existing opposing Defend
state. Rejected actions do not consume Defend or advance turns. The resolver
does not advance turns; Battle will connect resolution and accepted-action
completion in Milestone 8.

### Brace mechanic contract

Brace is an authored Branoc utility move:

```text
MoveKind.UTILITY
TargetType.SELF
ResourceType.MANA
resource_cost = 5
mechanic = "brace"
```

Brace is flavored as Branoc using Sunder-Spire defensively.

Brace is not a stat increase. It is temporary incoming physical damage reduction
plus a prepared heavy-attack payoff owned by CombatState when later implemented.

On accepted use, Brace must:

```text
spend Mana
activate CombatState-owned Brace state
reduce next eligible incoming physical damage by 40%
expire incoming protection after the next accepted opposing action, hit or miss
arm heavy_attack payoff
```

Rejected Brace attempts must:

```text
spend no Mana
activate no state
advance no turn/lifecycle state
```

### Incoming protection

Brace incoming protection:

```text
reduces eligible incoming physical damage by 40%
does not reduce magical or hybrid damage in 9A
expires after the next accepted opposing action, hit or miss
```

Expiry does not wait for a landed hit and does not wait for a physical action.
The next accepted opposing action consumes the protection window even if that
action is magical, hybrid, or a miss.

Brace activation itself must not expire its own protection.

Required lifecycle meaning:

```text
Branoc uses Brace
Brace state activates
Branoc's accepted Brace action completes
Brace incoming protection remains active
Enemy takes an accepted action
Brace incoming protection expires after that accepted enemy action
```

Do not implement expiry as "any accepted action after Brace."

### Heavy payoff

Brace arms a follow-up payoff for the next accepted move with:

```text
mechanic == "heavy_attack"
```

Heavy payoff behavior:

```text
next accepted heavy_attack gains +30% damage
payoff consumes on accepted heavy_attack use, hit or miss
rejected heavy_attack attempts do not consume payoff
accepted nonmatching Branoc moves do not consume payoff
```

The payoff should be consumed before or during hit resolution for an accepted
heavy_attack so that an accepted miss still consumes it.

### Refresh / no stacking rule

Repeated accepted Brace uses refresh or replace the existing Brace state for that

Brace must not stack:

## M10 — Overworld Route and Multi-Enemy Combat

M10 should establish the surface route leading toward the dungeon and Goblin Lord.

The game loop becomes:

```text
character selection
→ overworld route
→ encounter
→ victory rewards
→ rest, equip, progress, or save
→ next encounter
→ dungeon entrance
```

This is no longer just a second Goblin fight. It is the first real session loop.

### Encounter Route

The intended escalating route is:

```text
Goblin
→ Goblin pair
→ Goblin Warrior
→ Goblin Warrior pair
→ Goblin Shaman
→ Goblin Shaman pair
→ Goblin Elite with Goblin support
→ Goblin Lord with Goblin and Goblin Warrior support
```

In encounter-count form:

```text
1 Goblin
2 Goblins
1 Goblin Warrior
2 Goblin Warriors
1 Goblin Shaman
2 Goblin Shamans
2 enemies: Goblin Elite + Goblin
3 enemies: Goblin Lord + Goblin + Goblin Warrior
```

The fixed route should communicate:

```text
familiar enemy
→ increasing enemy count and pressure
→ stronger individual threats
→ mixed enemy roles
→ reinforced boss finale
```

### Multi-Enemy Combat

The first multi-enemy gate should be:

```text
one player character
→ two Goblins
→ select either target
→ both enemies can act
→ defeating one does not end combat
→ defeating both ends combat
```

Required architecture:

- Exact target identity for every action and status
- Enemy-specific HP, actions, and temporary state
- Target selection in semantic input
- Multiple enemy panels in presentation
- Victory only when all enemies are defeated
- Defeat when the player is defeated
- Enemy turns routed individually
- No status leakage between enemies
- Gravemantle Break remains linked to one target
- Frost, Burn, Poison, Frozen, and Stun remain target-specific
- Overcharge may discharge against a different target while clearing its linked setup
- No multi-target moves yet unless explicitly approved

The existing identity-safe status architecture gives us a good foundation for this expansion.

- A full map
- Quest tracking
- Shops
- Dialogue trees
- Multiple chapters
- Procedural exploration

Resting must have an explicit contract. The initial choice should be decided before implementation:

```text
full HP and Mana
```

or:

```text
partial recovery with a resource cost
```

Resting should occur between encounters, not during combat.

## M10B — Encounter Route

Add the fixed Goblin route as authored encounter data.

Each encounter should define:

- Enemy composition
- Encounter identifier
- Reward values
- Available rest boundary
- Next route destination
- Whether the encounter is a boss encounter
- Completion state

The route should be deterministic and testable.

The Goblin route must not modify production enemy balance merely to compensate for the new system.

## M10C — Multi-Enemy Combat

Expand the battle model to support multiple enemies without rewriting Battle ownership boundaries.

Battle remains orchestration.

CombatResolver remains responsible for action mechanics.

CombatState owns encounter-local state.

Required tests:

- Two enemies can coexist
- Target selection is exact
- One enemy can die while the other remains active
- Enemy responses continue correctly
- All enemies must be defeated for victory
- Simultaneous player and enemy death remains defeat
- Defeated enemies take no actions
- Status cleanup is target-specific
- Frozen and Stun suppress only their affected target
- Burn, Poison, and Frostbite ordering remains deterministic
- Gravemantle Break never affects another enemy
- Presentation shows each target’s independent states

## M10D — Equipment Integration

Connect existing equipment to effective combat values.

Required behavior:

```text
base character values
+ equipped item modifiers
= effective combat values
```

Equipment must affect:

- Effective attack values
- Effective defense values
- Relevant stat-derived damage or mitigation
- Maximum HP and Mana where supported

Equipment must not mutate permanent base attributes.

Required gate:

```text
create player
→ record combat values
→ equip item
→ observe changed effective values
→ unequip item
→ observe values return
```

Also prove:

- Equipment belongs to the correct PlayerState
- Two sessions do not share equipment mutations
- Equipment remains present across encounters
- Equipment state appears in snapshots
- Temporary combat state does not become equipment state

## M10E — Progression and Rewards

Connect victory to persistent progression.

After victory:

- Award EXP
- Preserve EXP in PlayerState
- Preserve current HP and Mana according to the encounter contract
- Award gold or existing persistent resources where supported
- Preserve level state
- Permit controlled permanent attribute growth
- Safely increase maximum HP and Mana
- Do not add skill trees yet

The progression API should be narrow and validated.

It should support:

- Reading level and EXP
- Applying approved EXP rewards
- Checking level thresholds
- Applying permanent growth through a controlled method
- Updating derived maximum resources safely

No direct mutation of arbitrary stat dictionaries should be exposed.

Required gate:

```text
win a fight
→ observe persistent EXP
→ observe persistent resource state
→ leave combat
→ enter the next encounter
→ confirm progression remains
```

## M10F — Resting

Resting is the bridge between combat and persistence.

The final rest contract should define:

- What resources recover
- Whether temporary states are already cleared
- Whether rest costs gold, time, or nothing
- Whether rest is always available
- Whether resting can occur after the final boss
- Whether resting affects story progression

Initial implementation should remain simple and deterministic.

Resting must never:

- Restore encounter-local statuses inside an active battle
- Duplicate EXP
- Reset equipment
- Reset prepared run inventory
- Refill consumables unexpectedly
- Alter authored character data

## M10G — Save and Load

Save/load becomes meaningful once the route contains persistent state.

The save boundary should include:

- Character profile identity
- PlayerState
- Level and EXP
- Current HP and Mana
- Equipment
- Gold
- Run inventory
- Prepared Infusion state
- Route position
- Completed encounter identifiers
- Rest state where relevant
- Story/world progression

It must exclude:

- Live resolver objects
- Battle UI
- Presenter models
- Open menus
- RNG objects
- Enemy runtime references
- Object identity pointers
- Temporary encounter statuses

Snapshots must remain serializable and deterministic.

Save/load does not need:

- Save slots
- Cloud saves
- Autosave
- Version migration
- Multiple chapters

Those can follow after the first reliable save boundary exists.

## M11 - Session Integrity and Regression

M11 is complete and sealed. It proved the complete session object graph rather
than repeating isolated component tests.

Accepted coverage includes:

- canonical profile, Character, PlayerState, and GameState ownership
- persistent combat, progression, inventory, equipment, and run state
- encounter-local cleanup and multi-enemy target isolation
- defeat, Retry, Rest, rewards, and route progression
- all four Drifters completing the authored surface route
- save, exit, load, and continuation through the remaining campaign
- turn-scoped logs and inactive final Battle views
- final arrival at Dungeon Entrance after the Goblin Lord

M11 does not enter the dungeon or create companion or party state.

## M12 - v0.3 Documentation and Release

M12 closes v0.3 without changing production behavior. M11 already provides the
complete automated session acceptance proof, so M12 is limited to:

- updating the README and release authority
- documenting supported and deferred behavior
- running the existing full regression and compilation checks
- verifying exact-commit CI
- tagging the accepted release commit as `v0.3`

Compatibility cleanup, architecture restructuring, additional campaign tests,
and manual four-Drifter replays are deferred. They are not release blockers.

The v0.3 endpoint is exactly:

```text
Goblin Lord defeated
-> final rewards applied
-> Dungeon Entrance displayed
-> surface route complete
-> no dungeon action available
-> end of v0.3
```

- Skill trees

- Advanced crafting

- Multiple chapters
- Full quest systems
- Advanced enemy AI

- Final equipment catalogue

## Post-v0.3 Dungeon and Party Work

Entering the dungeon begins a later architectural phase. It is not implemented,
tested, or exposed as an action in v0.3.

Future dungeon and party work may begin only after the solo surface route has
proven:

- Progression persists
- Equipment matters
- Resting matters
- Saving works
- Multi-enemy combat works
- The Goblin Lord route is completable
- All four Drifters remain mechanically distinct

That later phase may introduce:

```text
solo PlayerState
-> dungeon entry
-> companion discovered
-> two-character party state
-> ally targeting
-> shared encounter routing
-> party-specific UI
```

The companion should not be added before the solo route and multi-enemy foundation are stable. Otherwise, party state, equipment, progression, and encounter routing all become entangled at once.
