# Dungeon Drifters v0.3 Milestones

## v0.3 Theme

**Unified Session State and Character Combat**

v0.3 should establish the architecture that future progression, equipment, enemies, story branches, and saving will build upon.

The release is complete when:

* one `GameState` owns the active session
* One `PlayerState` owns the persistent Drifter state
* Battle uses structured moves
* mana and stats have real mechanical effects
* player and enemies use a shared combat-facing contract
* All four Drifters have functional, distinct combat kits
* The state can be inspected through a serializable snapshot
* The existing vertical slice remains playable

# Milestone 1 — Establish the Session Root

Implement the top-level runtime object that represents one playthrough.

### Add

```text
GameState
├── PlayerState
├── StoryState
└── WorldState
```

### `GameState`

Owns:

* the selected player
* session metadata
* story progress
* world progress

### `StoryState`

Initially tracks only:

* current chapter
* current scene
* current location
* story flags
* completed events
* player decisions

### `WorldState`

Initially tracks only:

* discovered locations
* defeated encounters
* opened or consumed world objects
* persistent dungeon changes

### Main-loop flow

```python
character = events.pick_character()
player_state = PlayerState(character)
game_state = GameState(player_state)
```

The main loop owns the same `GameState` instance until the session ends.

### Gate

```text
Exactly one GameState is created per run.
The existing story and battle still work.
GameState contains the same PlayerState used by Battle.
No player data is copied into GameState.
```

# Milestone 2 — Define State Ownership and Inspection

Turn the existing object graph into an explicit contract.

### Persistent ownership

`PlayerState` remains authoritative for:

* Character
* current and maximum health
* current and maximum mana
* level
* EXP
* permanent attributes
* inventory
* equipment
* gold
* learned or available moves

### Temporary ownership

`CombatState` remains authoritative only for:

* turn count
* defending
* temporary buffs
* temporary debuffs
* temporary statuses
* stagger or prone state
* encounter-specific character resources
* temporary elemental modes

### Add state inspection

Implement:

```python
player_state.snapshot()
game_state.snapshot()
```

Snapshots return only:

* dictionaries
* lists
* strings
* numbers
* booleans
* `None`

Add:

```python
STATE_SCHEMA_VERSION = 1
```

### Do not implement yet

* writing save files
* loading save files
* save slots
* autosaving
* JSON migration

This is an **in-memory save-state foundation**, not yet a save system.

### Gate

A snapshot can show:

```text
chosen character
archetype
profile identity
attributes
current/max HP
current/max mana
level
EXP
gold
inventory
equipment
story location
story scene
story flags
```

Snapshot creation must not mutate the game.

# Milestone 4 — Formalize Stat and Resource Layers

Define how starting values become permanent and effective values.

## Attribute layers

```text
Starting value
    ↓ initializes
Permanent/base value
    ↓ equipment modifiers
Equipped value
    ↓ temporary modifiers
Effective combat value
```

Example:

```text
Starting Strength:       7
Permanent Strength:      9
Equipment modifier:     +3
Temporary debuff:       -2
Effective Strength:     10
```

### Required attributes

* Strength
* Constitution
* Intelligence
* Dexterity
* Charisma

### Starting values

Starting values define where the character begins. They are not caps and are not guaranteed effective minimums.

### Resource model

`Health` and `Mana` require mutable maximums:

```python
health.current
health.maximum

mana.current
mana.maximum
```

Add controlled methods such as:

```python
increase_maximum()
decrease_maximum()
heal()
take_damage()
restore()
spend()
can_afford()
```

### Initial stat responsibilities

```text
Strength
→ physical scaling and stagger pressure

Constitution
→ defense and durable techniques

Intelligence
→ magic, healing, and mana-related scaling

Dexterity
→ ranged attacks, accuracy, initiative, critical behavior

Charisma
→ luck, narrative checks, unusual influence mechanics
```

Charisma does not need to affect every combat formula in v0.3.

### Equipment rule

Equipment modifies effective values. It does not permanently rewrite base attributes.

### Gate

Tests must prove:

* permanent stats can increase
* Temporary modifiers do not alter permanent stats
* Equipment bonuses disappear when equipment is removed
* HP and mana maximums can grow
* current HP and mana remain clamped
* Effective stats resolve predictably

# Milestone 5 — Create a Shared Combatant Contract

Resolve the current player/enemy asymmetry before implementing the resolver.

The player currently has structured resources and state. Enemies currently have plain HP, mana, and move values. Both sides need the same combat-facing interface.

### Introduce `EnemyState`

```text
Enemy definition
├── starting attributes
├── starting resources
├── move loadout
└── identity

EnemyState
├── Enemy definition
├── Health
├── Mana
├── effective stats
└── encounter runtime state
```

### Shared combat contract

Both `PlayerState` and `EnemyState` should expose equivalent combat-facing behavior:

```python
display_name
health
mana_resource
combat_moves
effective_stat(name)
is_alive()
```

This may use a Python `Protocol`, abstract base class, or a carefully documented duck-typed contract.

### Remove from Battle

Battle should no longer own:

```python
self.foe_health
self.foe_max_hp
```

Enemy health belongs to `EnemyState`.

### Gate

The same resolver-facing code can read:

```python
combatant.health.current
combatant.mana_resource.current
combatant.combat_moves
combatant.effective_stat("strength")
```

for both player and enemy, without `isinstance` branches.

# Milestone 6 — Lock the Move and Resolution Contracts

Before calculating damage, define exactly what a move is and what resolving one returns.

## `Move`

Required fields:

```python
name
kind
resource_cost
power
scales_with
accuracy
target
damage_type
mechanic
description
```

`mana_cost` may remain temporarily, but `resource_cost` is more future-proof because some characters may eventually spend momentum, compounds, elemental strain, or other resources.

### Validate core values

Use enums or validated constants for:

* move kind
* scaling attribute
* target type
* damage type

The extensible `mechanic` identifier can remain string-based for specialized hooks.

## `MoveResult`

The resolver does not print. It returns a structured result:

```python
MoveResult(
    accepted=True,
    hit=True,
    move_name="...",
    resource_spent=5,
    damage=17,
    healing=0,
    statuses_applied=(),
    reason=None,
)
```

### Mutation rules to freeze

```text
Invalid menu input
→ does not consume a turn

Unknown move
→ rejected without cost

Insufficient resource
→ rejected without cost

Valid move attempt
→ cost is spent

Missed move
→ cost remains spent

Current HP and mana
→ persist after combat

Temporary combat effects
→ clear when combat ends

Equipment
→ affects effective values only
```

### Gate

Move validation and result creation work without requiring the interactive Battle loop.

# Milestone 7 — Implement the Core Combat Resolver

Create the mechanical engine used by Battle.

### Resolver responsibilities

1. Validate the move.
2. Validate the target.
3. Check resource availability.
4. Spend the resource.
5. Roll accuracy.
6. Resolve the scaling attribute.
7. Calculate raw damage or healing.
8. Apply defense or resistance.
9. Apply health changes.
10. Apply basic status hooks.
11. Return `MoveResult`.

### Formula philosophy

The first formulas should be:

* transparent
* easy to test
* easy to rebalance
* shared across characters
* free of character-specific branching

Do not bury named-character exceptions inside the generic resolver.

### Randomness

Inject the random-number source:

```python
CombatResolver(rng=random)
```

Tests should supply deterministic RNG behavior.

### First supported actions

* basic physical damage
* basic magical damage
* healing
* mana spending
* accuracy and misses
* defense mitigation
* self-targeting
* enemy-targeting

### Gate

The resolver can execute a complete combat action without:

* printing
* reading input
* knowing which character profile is active
* knowing which story scene is active

# Milestone 8 — Rewire Battle

Make Battle consume the systems built in the previous milestones.

### Battle remains responsible for

* starting an encounter
* initiative
* displaying available moves
* accepting input
* selecting targets
* enemy action selection
* turn advancement
* victory and defeat detection
* rendering `MoveResult`

### Battle no longer owns

* generic steady/heavy attacks
* hard-coded damage rolls
* generic player recovery
* global miss logic
* player mana calculations
* enemy HP copies
* move formulas

### Move menu

The player menu should read directly from:

```python
player_state.combat_moves
```

Each move should display at least:

```text
name
resource cost
short description or mechanical summary
```

The battle HUD should show:

```text
Player HP
Player mana or active resource
Enemy HP
Relevant temporary statuses
```

### Resolve the existing healing issue

Healing output must report **actual health restored**, not the rolled amount.

### Gate

The Goblin encounter is playable from beginning to end using structured moves and the resolver.

No legacy `moves` dictionary is used by player combat.

# Milestone 9 — Implement Character Combat Identities

Build one complete kit at a time.

Each Drifter needs:

* a basic, reliable action
* meaningful resource use
* at least one stronger or conditional action
* one signature mechanic
* a clear strength
* a clear limitation

## 9A — Ser Branoc

**Role:** Heavy Vanguard

Minimum identity:

* durable physical offense
* guard or brace interaction
* high Constitution value
* slow but reliable pressure
* one Sunder-Spire signature technique

Possible temporary mechanic:

```text
Guard / Resolve / Momentum
```

## 9B — Azhvielle

**Role:** Elemental Controller

Minimum identity:

* Intelligence-based elemental damage
* mana-heavy choices
* at least two elemental effects
* one self-targeted or battlefield-control spell
* risk or instability hook

Do not implement the full causal-sorcery narrative system yet. v0.3 only needs a functional mechanical representation.

## 9C — Zhaivra

**Role:** Alchemical Marksman

Minimum identity:

* Dexterity-based ranged attacks
* limited compounds or prepared charges
* accurate basic shot
* elemental or chemical payloads
* one setup-and-trigger interaction

## 9D — Joruun

**Role:** Elemental Skirmisher

Minimum identity:

* mobile physical/elemental actions
* wind, water, and lightning identity
* only one active current at a time
* switching or maintaining a current matter
* Physical or resource cost prevents unrestricted cycling

### Gate for each character

Before starting the next kit:

```text
The character can complete the Goblin encounter.
Every move has deterministic tests.
Resource use is enforced.
The signature mechanic works.
State survives the encounter correctly.
No other character regresses.
```

v0.3 is ready when this flow is real:

```text
CharacterProfile
        ↓
Character starting definition
        ↓
PlayerState persistent progression
        ↓
GameState session ownership
        ↓
Battle orchestration
        ↓
CombatResolver
        ↓
Structured MoveResult
        ↓
Persistent HP, mana, EXP, equipment, gold, and story state
```

And all four characters must feel mechanically different while using the same underlying architecture.
